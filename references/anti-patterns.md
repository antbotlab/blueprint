# Anti-Pattern Catalog

This catalog documents known failure modes in agent-driven construction planning. Use it in two ways:

1. **Author self-check (Phase 3)**: scan this list before finalizing the plan draft.
2. **Reviewer verification (Phase 4)**: independently verify the plan contains none of these patterns.

The duplication is intentional â€” author self-check and reviewer verification are separate safety layers. Do not remove either.

---

## Table 1: Planning Anti-Patterns

| # | Anti-Pattern | Why It Fails | Fix |
|---|---|---|---|
| 1 | **Mega-step** â€” a single step covers multiple logical changes (e.g., "Refactor auth module, migrate DB schema, and update all tests") | Too large to review as one PR; executing agent cannot complete it in a single branch without accumulating risky state; a mid-step failure leaves the codebase in an unshippable intermediate state | Split into steps where each has a single logical intent describable in one sentence and touches â‰¤15 files. Apply the one-PR-sized heuristic from Phase 2. |
| 2 | **Implicit cross-step dependency** â€” Step 5 silently reads an artifact (generated file, env variable, migrated schema) created by Step 3, but the dependency is not declared in the dependency graph | Parallelism decisions are made on incomplete information; if Step 5 is assigned to a parallel group or executed before Step 3, it fails in a non-obvious way that is hard to diagnose in a new session | List every artifact each step produces and consumes. Draw explicit edges in the dependency graph. If Step B needs output from Step A, add edge Aâ†’B. |
| 3 | **Vague exit criteria** â€” Verification section says "check that everything works" or "ensure tests pass" without naming which tests or what pass means | The executing agent cannot self-verify completion; reviewers cannot assess whether the step is truly done; re-entry after an interruption is ambiguous | Write concrete, machine-verifiable commands: `pnpm test --reporter verbose`, `gh pr checks`, `curl -s localhost:3000/health \| jq .status`. Every exit criterion must be a command the agent can run and compare against an expected output. |
| 4 | **Copy-paste verification** â€” every step's Verification section has the same command (e.g., `pnpm build && pnpm test`) regardless of what that step actually changed | Generic checks do not verify step-specific correctness; a step that added a new module could "pass" even if the module is never imported or exercised; the pattern also signals the author did not think carefully about each step's distinct success condition | Each step's verification must target what that step changed. A step adding a new API endpoint should verify the endpoint responds; a step migrating a DB schema should verify the schema matches the target. |
| 5 | **Premature strongest-model assignment** â€” every step is assigned to the strongest model regardless of whether it involves architectural decisions | The strongest model is rate-limited and slower; assigning it to mechanical tasks (file moves, test writing, config changes) wastes capacity and creates unnecessary bottlenecks without quality benefit | Reserve the strongest model for steps that meet at least one criterion: architectural decisions affecting 3+ modules, risk assessment for breaking changes, review/audit tasks, steps where a wrong decision is expensive to reverse. Default everything else to the default model. |
| 6 | **Plan-as-documentation** â€” steps are written as narrative prose ("We will then migrate the legacy auth service to the new OAuth flow, taking care to preserve backward compatibility") instead of imperative agent instructions | An agent executing the plan cold cannot extract concrete actions from narrative prose; the plan reads like a design document, not a work order; ambiguity in prose forces the agent to interpret intent, introducing variance | Write every step in imperative voice with explicit Tasks. Use verb-first bullets: "Create `src/auth/oauth.ts`", "Run `pnpm test:auth`", "Open PR targeting `main`". The reader is an agent, not a human stakeholder. |
| 7 | **Missing rollback strategy** â€” steps with irreversible side effects (DB migrations, published packages, remote pushes) have no documented recovery path | When such a step fails mid-execution, the executing agent has no guidance on how to restore a consistent state; the next agent session starts from an unknown intermediate state with no path forward documented | For every step with an irreversible action, add a Rollback field: the concrete commands or procedure to restore the prior state (e.g., `git revert HEAD`, migration down command, package deprecation command). If rollback is genuinely impossible, state that explicitly so the agent knows to treat the step as a point of no return. |
| 8 | **Hardcoded paths/values in plan template** â€” the plan uses literal values specific to the author's machine or project (e.g., `/Users/alice/projects/myapp`, `main` as the assumed branch name, `npm` when the project uses `pnpm`) | When a different agent or environment executes the plan, hardcoded values silently fail or produce wrong results; hardcoded branch names break on repos with a different default branch | Use placeholders resolved during Phase 1 research: `{project-root}`, `{default-branch}`, `{package-manager}`. The Phase 1 pre-flight step detects the default branch; steps reference `{default-branch}` throughout. |
| 9 | **Skipping Phase 4 review** â€” the planning agent judges the plan "simple enough" and skips the strongest-model review gate | The planning agent is the author; authors have blind spots; "simple" plans have produced failed executions when a dependency was missed or a step was under-specified; the review gate exists precisely because simplicity is not self-evident | Never skip Phase 4. If the strongest model is unavailable, fall back to the default model per the degradation protocol. If neither is available, mark the plan UNREVIEWED and warn the user. The review gate is mandatory regardless of perceived simplicity. |
| 10 | **Monolithic dependency graph** â€” all steps are declared serial (each depends on the previous) when some could safely run in parallel | Serial execution multiplies wall-clock time; an 8-step plan with no parallelism where 4 steps are independent takes twice as long as necessary; this is especially wasteful when steps involve waiting (CI runs, review turnarounds) | Analyze each pair of steps: do they touch overlapping files? Does one need output from the other? If neither, they can be parallel. Group independent steps into parallel sets in the dependency graph. Mark parallel groups explicitly: `Group A (parallel): Step 2, Step 3, Step 4 â€” no shared files, no output dependencies`. |

## Table 2: Rationalization Rebuttals

| Rationalization | Why It's Wrong |
|---|---|
| "Skip review, it's trivial" | Triviality is assessed by the same agent that wrote the plan â€” the author. Author-as-reviewer is the weakest possible review. Plans rated "trivial" have consistently produced the class of bugs that adversarial review catches: missing dependencies, ambiguous exit criteria, implicit assumptions. The review cost is one sub-agent call; the cost of executing a flawed plan is re-work, state corruption, and debugging across sessions. Trivial plans pass review quickly. Non-trivial plans get caught. |
| "I'll update the plan later" | Plans are the source of truth for executing agents. An agent that resumes execution in a new session reads the plan file, not the coordinator's memory. Deferring plan updates means the plan diverges from reality; a future agent executes stale instructions and produces incorrect results or silently skips work that was actually done. Update the plan immediately when intent changes â€” not after, not "later". |
| "One big PR is faster" | It is faster to create. It is slower to review, slower to debug if CI fails, and produces an all-or-nothing merge risk. A single PR that touches 40 files across 6 logical changes cannot be reviewed meaningfully â€” reviewers approve on faith. When CI fails on a mega-PR, bisecting the failure requires reasoning about all 6 changes simultaneously. Atomic PRs take slightly longer to open but failures are isolated, reviews are actionable, and the merge history is bisectable. |
| "No tests needed for refactors" | Refactors change structure without changing observable behavior â€” which is exactly the condition that tests verify. A refactor without tests has no automated verification that behavior was preserved; the only evidence is "it compiled and I checked manually." Manual spot-checks miss edge cases. The test suite is the behavioral contract; a refactor that cannot be covered by tests either changes behavior (by definition not a pure refactor) or was not fully thought through. |
| "The plan is obvious, no need to write it down" | Plans are not written for the planning agent â€” they are written for the executing agent that starts cold with zero context in a future session. What is "obvious" to the planning agent (which has full conversation history, codebase context, and design rationale in working memory) is not obvious to a cold-start agent that has only the plan file and CLAUDE.md. The plan file is the only channel through which the planning agent's knowledge transfers to the executing agent. Skipping it breaks that channel. |
| "Let me just start coding, I'll plan as I go" | This produces a plan retroactively assembled from git history â€” which is not a plan, it is archaeology. Decisions made without a dependency graph skip parallelism opportunities and introduce implicit cross-step dependencies that go undocumented. Without exit criteria written before execution, "done" is defined by fatigue rather than correctness. Without a Phase 4 review gate, design flaws are discovered during execution when they are expensive to fix rather than during planning when they are cheap. |
